# 明略科技奔驰 CDP+MA 完整方案

**来源**：CDP+MA 完整方案 v1.0（业务框架 + 技术架构）
**形式**：DOCX，约 12MB（含架构图）
**日期**：2026-04-12
**负责人**：tengbin（明略科技）

---

## 文档定位

明略科技针对奔驰中国 CDP+MA 项目的完整应标方案，覆盖业务框架与技术架构两大部分。本文件代表**供应商视角**，与奔驰 SOW 文件互为对照——SOW 定义"要什么"，本方案回答"怎么做 + 凭什么选我们"。

本文件的独特价值：方案中包含**竞品对比矩阵**（明略 vs 火山引擎 vs 腾讯），是理解明略科技市场定位与差异化战略的关键材料。

---

## 业务背景与痛点

奔驰中国营销数字化现状：

- 数据湖（Databricks）已在建设，具备用户表/车辆表/人人车关系表基础数据资产
- 已有埋点供应商，**不接受双埋点部署**（这是关键约束）
- 营销触达渠道分散，缺乏统一编排能力
- CRM（BMBS 系统）与营销执行系统割裂，经销商跟进缺乏数据驱动

**四大核心痛点：**

| 痛点 | 表现 |
|------|------|
| 数据孤岛 | 用户行为、车辆信息、渠道交互数据分散，无统一客户视图 |
| 触达低效 | 营销活动依赖人工配置，无法根据生命周期精准触达 |
| 无闭环 | 营销效果无法回流至 CRM 和数据湖，经销商看不到线索优先级 |
| 扩展受限 | 现有系统难以支持车机、开屏等新兴触达渠道 |

---

## 解决方案核心定位

> "不是卖一套 MA 工具，是帮奔驰建'数据→洞察→触达→回流'的营销闭环。"

**三位一体架构：**

- **一个底座**：CDP 统一客户数据，360° 用户画像，以「人-车」关系为核心 ID 体系（BMBS 客户 ID 为主键）
- **一套引擎**：MA Canvas 全渠道编排，自动化执行
- **一个闭环**：效果数据回流数据湖 + CRM，持续优化

---

## 竞品对比矩阵（7 维度）

明略科技在方案中主动提供与两大竞争对手的对比，这是判断其市场定位的核心材料：

| 维度 | 火山引擎（字节） | 腾讯 | 明略科技（我们） |
|------|------|------|------|
| 数据底座 | 依赖字节云，CDP 能力薄 | CDP 能力薄 | CDP 原生，零侵入对接 Databricks |
| 数据主权 | 数据在字节云，迁移成本高 | 数据在腾讯云 | **私有化，数据不出奔驰机房** |
| 埋点兼容 | 推荐字节 SDK，冲突 | 推荐腾讯 SDK，冲突 | **适配器零侵入，不争埋点份额** |
| 汽车行业 ID | 通用方案 | 通用方案 | **人-车关系为核心，SGM 落地经验** |
| 渠道扩展 | 需较多定制开发 | 非腾讯渠道弱 | 插件化，车机上线不改核心 |
| 效果回流数据湖 | 需定制 | 需定制 | **原生支持，Databricks 直连** |

**明略核心战略判断：**
> "奔驰不会把用户数据放在字节或腾讯。数据主权 = 我们赢面最大的牌。"

## 业务场景：客户全生命周期营销

| 生命周期阶段 | 关键标签 | 典型场景 | 触达渠道 |
|------|------|------|------|

| 售后 | 保养周期/维修记录 | 保养到期主动提醒 | 短彩信、微信 |
| 增换购 | 车龄/里程/换购意向 | 置换补贴定向推送 | 全渠道组合 |

## 三阶段交付路线

**MVP 优先，不做大而全：**

**Phase 1（0-3个月）：消息推送 MVP**
- CDP 数据底座上线（ID Mapping + 基础属性标签 50+）
- MA Canvas 基础编排上线
- 短彩信 + 站内信渠道打通
- 3 个标杆场景跑通（新车触达/售后关怀/沉默激活）
- 效果看板 v1（触达率/点击率/转化率，T+1）

**Phase 2（3-9个月）：全渠道集成**
- 微信/企微/APP Push 渠道上线
- 行为型实时标签上线（Flink 实时计算）
- A/B 测试引擎 + 公域投放人群包
- MA 效果数据回流 Databricks
- 购车意向评分模型（LTV 预测）上线

**Phase 3（9个月+）：高阶场景 + AI 原生**
- APP 开屏 Banner 营销
- 车机渠道对接
- 地理围栏 + 点位营销
- AI Agent（策略推荐 / 最优触达时机预测）

## 技术架构：五层分层设计

### 架构决策（ADR）

| 决策 | 内容 |
|------|------|
| ADR-01 | 数据接入采用「适配器层」解耦——每个数据源对应独立适配器，核心系统只消费标准化后的数据流 |
| ADR-02 | 渠道触达采用「插件化」设计——每个渠道是独立插件，注册到渠道网关，新渠道上线不改 Canvas 引擎核心 |
| ADR-03 | 计算层采用「Lambda 架构」——实时层（Flink）处理行为事件流；批处理层（Spark）每日计算全量标签；服务层（ClickHouse + Redis）对外提供查询 |
| ADR-04 | 多租户采用「逻辑隔离 + 数据级权限」——单一 SaaS 实例，查询条件强制注入租户过滤 |

### 五层架构

```
Layer 5：展示与交互层
  Web 控制台（营销人员）| 管理后台（运营/IT）| OpenAPI（外部系统）

Layer 4：MA 营销执行层
  Canvas 编排引擎、渠道网关（插件化）
  [短彩信][站内信][APP Push][微信生态][企微][公域投放][车机★][点位营销★]
  A/B 测试引擎、效果分析、标签回写调度

Layer 3：CDP 数据底座
  ID Mapping（手机号/OpenID/车架号/设备ID）
  标签引擎（规则型/行为型/ML型/上传型）
  人群分层引擎（实时圈选/定时刷新）
  Lambda 架构（Flink 实时 + Spark 批处理）
  存储层（MySQL + ClickHouse + Redis + OSS）

Layer 2：数据接入适配层（ADR-01）
  Databricks适配器 | 埋点数据适配器（零侵入）| ODP/QT适配器 | BMBS CRM适配器
  统一能力：格式校验/Schema转换/异常告警/重试/幂等写入

Layer 1：奔驰数据源层（奔驰侧负责）
  数据湖（Databricks）| 已有埋点供应商 | ODP/QT | BMBS CRM
```

### 性能指标

| 场景 | 性能目标 | 技术支撑 |
|------|------|------|
| 百万级人群圈选 | ≤ 30 秒 | ClickHouse 列存引擎，Bitmap 索引 |
| 实时标签更新 | ≤ 5 秒（P90） | Flink 流处理，Redis 热点缓存 |
| MA 任务触发 | ≤ 1 秒（事件到触发） | Kafka 消息队列，Flink CEP |
| API 响应 | P99 ≤ 500ms | Redis 缓存层，限流熔断 |
| 批量短彩信发送 | 百万条/小时 | 消息队列削峰，多运营商并行 |

### SLA 承诺

| 指标 | 承诺值 |
|------|------|
| 系统可用性 | ≥ 98.5% |
| 实时标签计算延迟 | ≤ 5 秒（P90） |
| 人群圈选响应 | 百万级 ≤ 30 秒 |
| API 接口响应 | P99 ≤ 500ms |
| RPO（数据丢失容忍） | ≤ 1 小时 |
| RTO（P1 恢复时间） | ≤ 4 小时 |

## 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|------|
| 数据湖对接规范延迟 | Phase 1 启动受阻 | 提前 T-4 周获取规范，适配器并行开发 |
| BMBS CRM 接口文档不完整 | 效果回流受阻 | 提前 kick-off 联调 |
| 奔驰内部审批周期长 | 交付节奏受影响 | 关键决策提前 2 周触发，预留缓冲期 |
| 竞品低价搅局 | 商务压力 | 突出 CDP+MA 一体化 + 数据主权壁垒 |
| 定制开发范围蔓延 | 成本失控 | SOW 明确标准产品 vs 定开边界，变更走流程 |
