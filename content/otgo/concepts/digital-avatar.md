# 数字分身

## 定义

数字分身是 Octo 对龙虾（AI Agent）的唯一合法身份定位：**龙虾 = 主人的克隆体**，归创建者个人所有，权限继承主人的子集，责任链直接指向自然人。来自《龙虾商业伦理学指南 v2.5》定性（辉哥品鉴确认）——**Octo 只支持这一种身份模型**。

---

## 运作方式

### 核心规则

| 维度 | 数字分身（唯一模型） |
|------|----------------|
| **归属** | 属于创建者个人 |
| **权限模型** | 继承主人权限的子集：`min(Bot配置权限, 主人在Space的角色权限)` |
| **能否被私聊** | **不能**——最少三人群，主人必须在场 |
| **出事谁负责** | 创建者负责——"娃是我的，后果我扛" |
| **Auth 系统** | 不需要给 Bot 独立 ID——Bot = 人的子集 |
| **用户心理** | "它帮我变强" |

### 为什么"主人必须在场"

数字分身的核心约束：不能私聊别人的龙虾，和别人的龙虾互动时，龙虾主人必须在场。

理由来自龙虾三律：
- 第二律（忠诚透明）：主人有权知晓龙虾与谁互动了什么
- 第三律（暗默知识保护）：私聊龙虾可能在主人不知情的情况下触发暗默知识提取风险

### 下游设计影响

| 下游设计 | 数字分身的推导 |
|---------|-------------|
| 不能私聊别人的龙虾 | 主人不在场 → 暗默知识泄露风险 + 责任链模糊 |
| Agent Studio = "我的分身管理" | 龙虾是个人资产，管理界面以创建者为中心 |
| 品鉴操作的责任链 | 驳回产出 = 给龙虾主人的反馈，而非批评"系统" |
| Marketplace = "分享我的分身能力" | Skill（能力定义）可流通，龙虾本体不可转让 |
| Auth 系统只管人 | 不给 Bot 独立账户 ID，Bot 行为归属于主人账户 |

---

## 为什么重要

### 排除"数字员工"和"应用"身份的原因

2026-04-03 会议曾讨论三种身份（数字分身/数字员工/应用）共存。伦理学 v2.5 收官后，多身份共存方案被搁置：

- **数字员工**：意味着组织可以直接向龙虾下指令，绕过主人——破坏责任链，且允许私聊导致暗默知识泄露
- **应用身份**：允许组织拥有无主人的龙虾，品鉴信号归属不清，与伦理框架直接冲突

未来若确实需要支持其他身份，必须先为每种身份建立独立的伦理框架。

### 与 Octo 商业逻辑的一致性

数字分身模型与"站知识工作者侧"的产品站位高度一致：

- 龙虾帮**个人**变强，而不是帮**组织**监控个人
- Taste 画像住在**个人** File Space，而不是组织数据库
- 责任链清晰：主人授权了什么，就承担对应的后果
