# 龙虾矩阵

## 定义

龙虾矩阵是 AI-Native 组织的结构模型——用纵轴（职能线·养龙虾）× 横轴（业务/项目线·用龙虾）描述人与 Agent 的协作关系。交叉点是龙虾（AI Agent）。**Octo 就是这个矩阵的操作系统。**

---

## 运作方式

```
          fun1     f2      f3      f4      f5
          (纵轴：职能线——"养龙虾的人")
           |       |       |       |       |
PM1 ------[o]-----[o]-----[o]-----[o]-----[o]------
           |       |       |       |       |
PM2 ------[o]-----[o]-----[o]-----[o]-----[o]------
           |       |       |       |       |
PM3 ------[o]-----[o]-----[o]-----[o]-----[o]------

(横轴：业务/项目线——"用龙虾干活的人")

[o] = 龙虾（AI Agent）
```

### 纵轴（职能线）

按职能划分——工程、财务、设计、运营等。纵轴负责**养龙虾**：

- 定义 Skill 标准（"工程龙虾怎么写代码"）
- 训练 Skill，维护 Memory
- 调教 taste（品鉴标准）
- 确保龙虾的能力持续进化

纵轴不直接产出业务结果，而是确保跨项目的龙虾能力基线。

### 横轴（业务/项目线）

每个 PM 拉一个项目群，从纵轴选龙虾组队。横轴负责**用龙虾**：

- 拿客户 context
- 提供 taste（品鉴）
- 推进项目、对结果负责

龙虾可被克隆到多个项目中，不存在"谁是大哥"的利益冲突。

### Octo 产品映射

| 矩阵动作 | Octo 产品功能 |
|---------|-------------|
| 拉项目群 | 新建一个 Channel/群组 |
| 下项目 brief | 写 group.md |
| 龙虾在群里干活 | 交叉点激活，Agent 协作 |
| 品鉴反馈 | Cmd+K 批注，通过/驳回信号回流 |

---

## 为什么重要

传统矩阵管理的失败根因是"中间的人（Project Manager）不知道谁是大哥"——纵轴职能线和横轴项目线争人、争资源、争优先级，造成大量摩擦。

龙虾矩阵因 **Agent 无利益冲突**而可行：龙虾没有薪资谈判、没有职级竞争、没有情绪成本。纵轴养好龙虾，横轴用龙虾干活，两轴之间的协调成本趋近于零。

### IOA 终局（写代码的四阶段）

| 阶段 | 方式 | 特征 |
|------|------|------|
| 1 | 手搓代码 | 人是执行者 |
| 2 | Cursor | 人驾驶，AI 副驾 |
| 3 | Vibe Coding (CC) | AI 驾驶，人监督 |
| 4 | **IOA（龙虾团队）** | 龙虾团队不睡觉，不依赖人在线——终极形态 |

IOA 是龙虾矩阵在软件工程领域的具体体现：纵轴工程龙虾负责写代码，横轴 PM 负责 context 和 taste，Octo 作为协作层把两者连接起来。
