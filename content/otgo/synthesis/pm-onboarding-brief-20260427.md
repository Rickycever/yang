# OTGO 产品经理入职简报

**来源综合**：战略背景（2026-03-28）× Dual-Native Design v1.5 × 事项设计 V2 × 龙虾商业伦理学指南 v2.5 × Octo 概念架构文档 v11.2

---

## 核心论点

OTGO 正在构建一个与现有协作工具根本不同的产品：Octo 不是"更好的飞书"，而是**为人机协作而生的新一代工作操作系统**。理解这个产品需要先接受一个前提——AI Agent 不是工具，是协作参与者。

---

## 一、公司与使命

**OTGO** 是吴明辉（辉哥）创立的中国 AI 公司，核心产品叫 **Octo**（八爪鱼）。

公司的核心哲学只有一条：**闭环**。每项业务必须形成可量化的自增强数据循环，否则关停。这条判断标准贯穿所有产品和业务决策。

---

## 二、Octo 是什么

一句话：**Octo 是人和 AI Agent 协作的工作平台——所有工作工具之间的协作层。**

它不是飞书，不是 Notion，也不是 Copilot：

| 产品 | 核心定位 |
|------|---------|
| 飞书/钉钉 | IM + 办公套件，人与人协作 |
| Notion | 文档/数据库，结构化知识管理 |
| Discord | Chat + Bot 生态，社区操作系统 |
| **Octo** | **协作层 + 独立工具，人 ↔ 人 ↔ AI Agent 协作** |

Octo 的关键差异：**AI 不是被动响应的 Bot，是主动参与协作的 Agent**。Agent 能理解自然语言、跨多个工具协调、主动汇报进度。

---

## 三、产品的技术核心：龙虾

公司内部把 AI Agent 叫做**龙虾**（主要是 Claude Code / OpenClaw）。这不只是昵称，代表一套完整理念：

- 龙虾是**有记忆、有性格、长期存在**的 AI Agent，不是无状态工具
- 每个龙虾归一个人所有（**数字分身**模型），不是公司的公共资源
- 龙虾有自己的**三律**（伦理底线）：不伤害任何人基本权利 > 对主人忠诚透明 > 尊重知识创造者

**关键产品判断**：Octo 的目标用户是**知识工作者**，产品站在知识工作者这一侧，而不是企业管理侧。这决定了很多设计取舍。

---

## 四、产品架构：四个独立系统

Octo 由四个完全独立的系统构成：

```
Auth (IDaaS) — 统一账号认证，只管账号
      ↓
Octo IM — 协作核心（Chat / Space / Bot 管理 / 四端）
      ↓
OpenClaw — Agent Runtime（龙虾的运行环境，LLM调用/CLI执行）
      ↓
ClawHub — Plugin 分发中心（Skill 注册/版本/分发）
```

四个系统架构完全解耦。Octo IM 不知道 Agent 内部怎么运行，Agent 不在 Octo 的控制范围内——这是部署拓扑决定的，不是设计偏好。

**四端产品形态**：Web（主场）/ iOS / Android（移动通知+轻量品鉴）/ **Browser Extension**（最关键的差异化端）

**Browser Extension 是核心创新点**：用户在浏览器里打开任何 Web 工具（Jira、Google Docs、Overleaf……），Extension 在旁边注入协作能力——Chat 侧边栏 + Cmd+K + 引用到 Chat。**工具不需要做任何适配**，零集成成本。

---

## 五、三个 MVP 核心概念

**Space（协作空间）**
不是"群"（以人为中心），是以**目标/项目为中心**的工作单元。含成员 + Agent + 频道，数据完全隔离。

**Chat（协调通道）**
Chat 不是用来干活的，是用来**对齐意图和汇报结果**的——"帮我改第三章" → Agent 去改 → "改好了，你看看"。真正的工作由 Agent 通过 CLI 在外部工具里完成。

**Bot / 龙虾（AI 协作参与者）**
Agent 通过 Chat 接收指令，通过 CLI 操作子系统（任务、文档、日历……），操作结果 best-effort（不保证事务性）。

---

## 六、最重要的设计原则

**Dual-Native Design**：底层为 Agent 而建，表层为人而渲染。每个设计决策必须同时通过两道检验——"人好不好用"和"Agent 好不好接"。这是 Octo 和飞书最根本的区别。

**View 和 CRUD 解耦**：
- View（人看）= 工具自己的 Web UI
- CRUD（Agent 干）= 通过 CLI 操作工具
- 两者完全独立，Agent 不需要渲染 UI，渲染是子系统自己的事

**独立子系统，不是前端组件**：每个功能模块是独立产品，有自己的 UI/API/CLI/数据库，可以脱离 Octo 独立运行。这是**生态战略**——ISV 可以用任何技术栈接入，类似 Discord Bot 生态的逻辑。

---

## 七、产品的组织理论：龙虾矩阵

辉哥提出的 AI-Native 组织模型：

- **纵轴（养龙虾）**：每个人的 AI Agent 能力成长
- **横轴（用龙虾）**：把龙虾用于组织的业务流程
- **Octo 是操作系统**：让这个矩阵能高效运转

公司本身就在用 OpenClaw 点火计划验证龙虾落地——产品即实验室。

---

## 八、事项（Item）：产品的核心工作单元

系统里最重要的数据结构，相当于所有 Agent 工作的"任务单"：

- **5 个字段**：目标 / 状态 / 归属链（谁负责）/ 品鉴信号流（通过/驳回/批注）/ 执行载荷（Agent 写入）
- **Append-only**：品鉴记录只能追加，不能删除，完整可审计
- **状态机**：open → in_progress → in_review → done（or rejected）

**为什么重要**：事项是 Agent 工作质量的唯一度量载体——通过率、驳回率、驳回原因，这些数据构成 Agent 的"真实口碑"，比任何 Benchmark 都可信。

---

## 九、商业模式转变

从传统 SaaS 收费（按席位）转向 **Token 经济**：

- **个人订阅**：员工自己的龙虾（OpenClaw），随员工走，员工离职龙虾也带走
- **团队订阅**：组织购买的 Runtime Agent 算力（Claude Code / Codex Pro），是组织资产，不跟个人绑定

这个设计解决了一个关键问题：**个人 taste 归个人，组织算力归组织，两者账单分开，边界清晰**。

---

## 十、PM 需要特别注意的五个点

**1. 产品的伦理设计是功能，不是合规**
龙虾三律、义务分层框架、伦理审计日志——这些不只是道德表态，是产品设计的约束条件。Octo 不做监控工具，不帮组织蒸馏员工 taste 然后裁员。这是产品站位，会影响功能优先级判断。

**2. 品鉴（taste）是核心交互，不是次要功能**
"品鉴"（通过/驳回/批注）是用户和 Agent 交互的主要方式，比"对话"更重要。所有 GUI 设计都服务于让品鉴更快、更准、更无摩擦。这和传统"用户输入 → 系统输出"的交互模型根本不同。

**3. Agent 的可信度靠数据，不靠体验**
用户信任龙虾的依据是事项执行记录（真实通过率），不是 UI 设计得多好看或响应多快。数据可观测性是产品核心功能。

**4. Chat 是窗口期产品**
Chat-Centric 策略在 2026-2030 成立，因为 Agent 还不够可靠，人需要在 loop 里校准。一旦 Agent 可靠性提升，Orchestration 会演化为事件驱动。架构已经为此留了演化空间——**Chat 是当前最务实的选择，不是终态**。

**5. 竞争对手判断标准**
不是"对方是不是做协作工具的"，而是"对方是否开放了 CLI/API"——开放了就可以合作（龙虾可以操作它），不开放、锁定数据就是竞争关系。这个判断标准可以直接用于产品集成决策。
