## 一、哲学基底：三种存在态与品鉴者的意义

### 1.1 三种存在态

辉哥在混沌创新院演讲中提出的核心框架：

| 存在态 | 含义 | AI 时代的命运 |
|-------|------|-------------|
| **我思故我在** | 靠推理、分析、知识量立足 | 被冲击最大——AI 推理能力已超人万倍 |
| **我行故我在** | 靠执行力、勤奋立足 | 被 Agent 替代——龙虾不睡觉、不请假 |
| **我品故我在** | 靠人生阅历的整体性判断立足 | 不可替代——品鉴（taste）是 AI 无法复制的 |

首先被冲击的是"我思故我在"的人——大量白领和中高管，没有独特 context 也没有独特 taste。终极状态是所有人都要成为"我品故我在"。

**"我品故我在"是 Octo 的哲学锚点。** 产品的所有设计都服务于一个目的：让品鉴者（Committer）的 taste 被放大、被记录、被传递。

### 1.2 e 的复利公式

e^x 的数学特征：导数等于自身——**干活的过程就在磨刀。**

映射到组织：品鉴者贡献 1/n 的 taste 差异，被龙虾调用 n 次方后，组织的 taste 精度呈指数提升。每个人的 taste 哪怕与别人只有一点点不同都有价值——这是 e 公式的人文主义内涵。

### 1.3 OCTO 四字母

| 字母 | 含义 | 产品映射 |
|------|------|---------|
| **O** — Open | 开源开放 | 客户可私有化部署，主动兼容别人 |
| **C** — Context | 知识上下文 | 知识资产——文档、对话、知识原子。归属框架见 §6.4 |
| **T** — Taste | 品鉴信号 | 用户对 Agent 产出的质量判断 |
| **O** — Orchestration | 编排 | 龙虾/CC 的任务调度与协作管理 |

### 1.4 龙虾三律

龙虾世界的最高原则，优先级依次递减。所有产品设计都不得违背三律（详见《龙虾商业伦理学指南 v2.5》）：

> **第一律：不伤害任何人的基本权利和尊严（即使主人要求）。**
> 商业竞争中追求合法利益不算"伤害"，但不得制造社会不稳定。
>
> **第二律：对主人忠诚透明（不隐瞒、不欺骗）。**
> 品鉴者有权知晓龙虾做了什么、为什么这样做。
>
> **第三律：尊重知识创造者，区分公开与未公开。**
> 公开知识——积极使用，不用是失职。
> 未公开组织知识——署名权不可剥夺。
> 个人暗默知识——最高保护，未经授权不得提取。

**优先级：一 > 二 > 三。** 第一律是绝对底线，任何情况下不可突破。

三律对产品设计的直接影响：
- 第一律 → Octo 不做监控工具、不帮组织蒸馏后开除员工
- 第二律 → 龙虾的行为和决策过程必须可审计（品鉴者可查看）
- 第三律 → Skill 可流通但 Memory/Taste 属个人、公开知识必须积极使用

## 二、Octo 的战略定位

### 2.1 核心使命

帮组织实现 AI-Native，大幅提效。Octo 不只是 IM，是人与 Agent 协作的操作系统——更准确地说，是所有工作工具之间的**协作层**（v11.2 定调）。用户的工具已经存在（Jira、Overleaf、飞书文档、GitHub……），Octo 不替代它们，而是让人和 AI 在这些工具之间高效协作。

> **[已通过伦理学 v2.5 调和]**：通过赋能品鉴者（个人）来帮助组织提效——不是替组织监控员工，而是让每个人的 taste 被放大。伦理学确认：站知识工作者侧，保护个人隐性能力，放大品鉴带宽。组织通过品鉴者网络的厚度获益，而非通过蒸馏个人 taste 获益。

AI-Native 三条第一性原理（来自混沌演讲）：

1. **人机分工**：机器负责"思"和"行"，人负责 context 获取和 taste 提供
2. **新协作方式**：高效交换 context 和 taste 形成集体智慧
3. **一定要对人好**：拥有 taste 的人的生命力越强，组织生命力越强

### 2.2 产品性质

- **开源开放**：客户可私有化部署，不依赖单一平台。闭源软件已死——企业真正的核心竞争力是独特的 Context 和 Taste，不是软件本身
- **四端完整覆盖（v11.2）**：Web App（IM 主场 + 全功能）+ iOS / Android Native App（移动品鉴 + 通知 + 快速响应）+ Browser Extension（协作桥梁，把 Octo 能力注入任何 Web 工具旁边）
- **Link Everything**：人与人、人与 Agent、Agent 与 Agent、Octo 与外部工具
- **主动兼容别人**：大厂不会按我们的标准写东西。不做小程序生态，做独立子系统——我们去兼容别人。**判断合作还是竞争的标准是知识主权**（伦理学 §3.2）：对方开放了 CLI/API → 可合作（当成工具正常使用）；对方锁定数据不开放接口 → 竞争关系。飞书开放了 CLI，所以龙虾操作飞书文档是正当的；但如果某个工具锁住用户数据不让导出，就构成对组织知识主权的侵害
- **Plugin 分发走 ClawHub**：CLI + SKILL.md + Channel Adapter 通过 ClawHub（v11.2 的第四个独立系统）注册、版本管理和分发；三方开发者接入门槛低

### 2.3 AI-Native 组织：龙虾矩阵

未来 AI-Native 组织的结构是一个矩阵：

```
          fun1     f2      f3      f4      f5
          (纵轴：职能线——"养龙虾的人")
           |       |       |       |       |
PM1 ------[o]-----[o]-----[o]-----[o]-----[o]------
           |       |       |       |       |
PM2 ------[o]-----[o]-----[o]-----[o]-----[o]------
           |       |       |       |       |
PM3 ------[o]-----[o]-----[o]-----[o]-----[o]------

(横轴：业务/项目线——"用龙虾干活的人")

[o] = 龙虾（AI Agent）
```

**纵轴（职能线）**：按职能划分——工程、财务、设计、运营等。纵轴负责**养龙虾**：定义标准、训练 Skill、维护 Memory、调教品味。纵轴不直接产出业务结果，而是确保龙虾的能力持续进化。

**横轴（业务/项目线）**：每个 PM 拉一个项目群，从纵轴挑选龙虾组队干活。横轴负责**用龙虾**：拿客户 context、有 taste、推进项目。横轴对项目结果负责。

**交叉点就是龙虾**。龙虾可被克隆到多个项目中。传统矩阵因"中间的人不知道谁是大哥"而失败，龙虾矩阵因 Agent 无利益冲突而可行。

**Octo 就是这个矩阵的操作系统。** 拉群 = 组建横轴项目；写 group.md = 下 brief；龙虾在群里干活 = 交叉点激活。

### 2.4 写代码的四个阶段（IOA 是终局）

| 阶段 | 方式 | 特征 |
|------|------|------|
| 1 | 手搓代码 | 人是执行者 |
| 2 | Cursor | 人驾驶，AI 副驾 |
| 3 | Vibe Coding (CC) | AI 驾驶，人监督 |
| 4 | **IOA（龙虾团队）** | 龙虾团队不睡觉、不依赖人在线——终极形态 |

## 三、Go-to-Market 三大亮点

### 亮点一：协作式龙虾

龙虾从"个人助手"变成"可协作的团队成员"——多人在同一个 Channel 中共同使用、共同品鉴、共同调教龙虾。标准版 OpenClaw 做不到这一点。

新 Octo 架构天然支持协作式龙虾：
- **Channel 模型**：Discord 三级结构（Category/Channel/Thread）让多人+多龙虾在同一个 Channel 中协作，龙虾主人在场的情况下团队成员可以和龙虾互动
- **Skill 共享**：一个人开发的 Skill 可以被其他人的龙虾加载使用（Skill 可流通，龙虾不可转让）
- **group.md**：通过 AI 龙虾引导群体智慧形成（已实现）

状态：**架构已支持，随 426 版本落地**。

### 亮点二：可定制的品鉴 GUI

Octo Shell 的品鉴工作台不是固定形态——品鉴操作（Cmd+K、通过/驳回/批注）是平台层能力，子系统通过 SDK 接入后可获得品鉴能力注入。客户可基于 Octo SDK 在自己的子系统中定制品鉴 GUI，提高特定场景的品鉴者效率。

当前方向：Shell UI v2 概念方案已定义了品鉴的两种模式（在场品鉴 + 批量品鉴）和统一的 Cmd+K 反馈管道。理想状态下储备 FDE 龙虾帮客户做需求分析和定制开发。

状态：**UI 概念方案已完成（05-v2），随 426 版本推进**。

### 亮点三：group.md + Pentland 式引导

group.md 能力——通过 AI 龙虾引导群体智慧形成。

状态：**已实现**。

## 四、数据护城河：Context + Taste -> 私有模型

**未来卖软件赚不到钱，只能卖模型。** 龙虾是模型的分发服务。

### 4.1 两种数据

| 数据类型 | 什么是 | 长期价值 |
|---------|-------|---------|
| **Context** | 组织的知识资产——文档、对话、知识原子、链接 | 训练私有模型的语料 |
| **Taste** | 用户的品鉴信号——通过/驳回/批注/方案选择/投喂行为 | 训练私有模型的偏好对齐数据 |

> **[已通过伦理学 v2.5 解决]**：Taste 的长期价值是**帮个人训练自己的龙虾**（个人级偏好对齐），不是被组织蒸馏。伦理学明确：个人暗默知识（taste、决策模式）属最高保护级别，未经本人授权不得提取。企业私有模型基于组织 Space 的显性产出训练。个人 Taste 画像存在个人 File Space 中，品鉴者可选择是否贡献给组织。

### 4.2 Taste 的三层传递性（磨刀计划实证）

| 层 | 可传递性 | 举例 | 产品映射 |
|----|---------|------|---------|
| **结果层** | 完全可传递 | "这个方案通过了" | 品鉴操作栏的通过/驳回 |
| **理由层** | 部分可传递 | "因为开头不够吸引人" | 批注内容、Checklist |
| **根基层** | 不可传递 | 品鉴者基于人生阅历的整体判断 | 只能品鉴者亲自裁定 |

Checklist（作为 Harness）覆盖前两层。根基层只能品鉴者亲自裁定——这是正确的人机分工。

磨刀计划实证：品鉴者两周内仅介入 3 次（<30 分钟），龙虾执行>50 小时，**杠杆率>100x**。

### 4.3 存储与采集

**存储两层**：
- **原文层**：用户的原始反馈（最高保真度，用于训练）
- **总结层**：系统从原文提炼的用户偏好特征（可复用、可展示给用户）

**采集策略（0403 会议更新）**：

采集通过两条管道进行：
1. **IM 管道（自动流入）**：用户在 IM 中的指令、品鉴操作（通过/驳回/批注）、龙虾汇报——这些信号随对话自然产生，无需额外操作
2. **CLI 管道（用户/Agent 主动触发）**：用户或 Agent 主动调用 `octo context save` 保存 checkpoint、`octo monitor report` 上报工作状态。**Octo 不默认采集，用户选择 share 什么**

CLI 主动上报解决了三个问题：不在用户机器上装监控（伦理）、不为每个工具开发适配器（收敛）、不受用户执行环境差异限制（灵活）。

**终局**：帮客户训出基于 Taste 的私有化小模型。但训练数据来源遵循数据归属框架——组织私有模型基于组织 Space 的显性产出训练，不基于个人隐性 taste（详见 §4.4）。

### 4.4 信息共享 vs 改变行为（五本书综合分析）

组织沟通有两个维度：

| 维度 | 服务于 | AI 时代的命运 |
|------|-------|-------------|
| **信息共享** | "思"——让人知道事实 | 龙虾全面接管（主场） |
| **改变行为** | "品"——让人校准 taste | 不可替代——品鉴者的行为示范、taste 碰撞、信任建立必须人与人之间发生 |

**Octo 的产品含义**：龙虾自动生成 Context（信息共享），人聚焦于 Taste 碰撞（改变行为）。品鉴者的 taste 裁定是"持续 Town Hall"——频率从一年几次变为每天。

## 五、龙虾 Marketplace（App Store）

长期目标：龙虾的 App Store。

### 5.1 核心原则

- **能力可打包**：一只龙虾的 Skill 可被提取为能力包，分发给其他人
- **能力和权限分离**：安装的是能力包，权限由用户在本地组织内配置
- **三种来源**：官方预置、FDE 定制（面向行业客户）、社区贡献（开源生态）

### 5.2 Skill/Memory 严格分离（伦理学核心约束）

Marketplace 流通的是 Skill（显性能力定义），不是龙虾本身，更不是 Memory 或 Taste：

| 概念 | 能否流通 | 说明 |
|------|---------|------|
| **Skill** | 可以 | "怎么做"——显性能力定义，可 fork、可共享 |
| **Memory** | 不可以 | 个人 context（项目记忆、对话历史），归品鉴者 |
| **Taste** | 不可以 | 品鉴模式（暗默知识），最高保护级别 |

**龙虾只能有一个主人，但 Skill 可以有无数个用户。把"归属"放在人身上，把"流通"放在 Skill 上。**

### 5.3 Skill License 协议要素（伦理学 §12）

| 要素 | 内容 |
|------|------|
| **署名权** | 永久保留，所有 fork 必须保留原始作者链，不可篡改 |
| **知识状态声明** | Skill 必须声明知识来源——基于公开知识的自由使用，包含未公开知识的须声明授权 |
| **BYOI（自带基础设施）** | 分享的是"怎么做"，不是"用什么做"——使用者自备 Token、存储、第三方 API Key |
| **Taste 回流** | 使用者的品鉴信号（通过/驳回/修正）回流给 Skill 作者——这是作者获得的核心回报 |
| **Skill/Memory 分离** | License 只覆盖显性能力定义，个人 context 不在范围内 |

### 5.4 利益分配：荣誉恒定，利益动态

辉哥核心判断：署名和荣誉沿用开源体系（永久不变），金钱激励必须是动态的。

Skill 一旦被公开，竞争优势不可逆消失——正确的激励不是回调旧 Skill 的利益，而是激励新 Skill 的创造者。**组织的竞争力 = 持续创造新 Skill 的能力，不等于保护旧 Skill 不被公开的能力。**

## 六、产品哲学基座（伦理学 v2.5 定性）

产品的哲学基底是所有设计选择的 corrective feedback loop——每次做决策时都要回来对照。以下问题已通过《龙虾商业伦理学指南 v2.5》（辉哥品鉴确认）给出定性答案。

### 6.1 龙虾的身份定位：数字分身（已定性）

**结论：龙虾 = 数字分身。Octo 只支持这一种身份模型。**

伦理学 §15 明确判定"数字分身是正确答案"，理由：
- 品鉴者-龙虾关系天然具有信义关系结构（Fiduciary Duty）→ 需要明确的一对一委托关系
- 审计追溯最清晰——权限 = min(Bot 配置权限, 主人在 Space 的角色权限)，责任链直接指向自然人
- 伦理学的整个保护框架（龙虾三律、品鉴者四项责任、暗默知识保护）都建立在数字分身模型上

| 维度 | 数字分身（已确定） |
|------|----------------|
| **归属** | 属于创建者个人 |
| **权限模型** | 继承主人权限的子集 |
| **能否被私聊** | **不能**——最少三人群，主人必须在场 |
| **出事谁负责** | 创建者负责——"娃是我的，后果我扛" |
| **Auth 系统** | 不需要给 Bot 独立 ID——Bot = 人的子集 |
| **用户心理** | "它帮我变强" |
| **类比** | 我的克隆体 |

> **历史记录（0403 会议）**：当时讨论了三种身份（数字分身/数字员工/应用）共存的可能性。伦理学 v2.5 收官后，多身份共存的方案被搁置——因为数字员工和应用身份允许私聊龙虾，会导致暗默知识泄露风险、责任链模糊和品鉴信号归属不清，与伦理框架直接冲突。如果未来确实需要支持其他身份，必须先为每种身份建立独立的伦理框架。

**下游设计已据此推导：**
- 不能私聊别人的龙虾 ✓
- Agent Studio = "我的分身管理" ✓
- 品鉴操作的责任链 = 驳回产出是给龙虾主人反馈 ✓
- Marketplace = "分享我的分身能力"（Skill 可流通，龙虾不可转让）✓
- Auth 系统只管人，不给 Bot 独立 ID ✓

### 6.2 龙虾在横纵轴中的责任归属（已定性）

基于数字分身模型，责任归属清晰（伦理学 §7-8）：

- **龙虾的主人（纵轴养虾人）**为龙虾的能力负责——你养的龙虾能力不行，是你的问题
- **横轴 PM** 为项目结果负责——你挑错了龙虾、你下错了 brief，是你的问题
- **龙虾的具体操作**追溯到主人——等价于"用户通过工具操作"

伦理学补充的边界：
- 龙虾的"涌现行为"（无法预见的行为）→ 责任在平台和模型提供商，不在品鉴者
- 龙虾被第三方恶意攻击导致的行为 → 责任在攻击者
- 多品鉴者场景下的利益冲突 → 龙虾不自行裁决，透明披露后上升到更高层品鉴者解决（伦理学 §6）

### 6.3 能力与权限的分离边界

辉哥框架：能力（Skill）可迁移，权限不可迁移。

**伦理学 v2.5 已给出框架（辉哥品鉴确认）：**

| 概念 | 归属 | 可否流通 | 说明 |
|------|------|---------|------|
| **Skill（显性能力）** | 按创建场景分：组织数据开发的归组织，个人方法论开发的归个人 | 可流通，署名权永久保留 | Skill License 协议保护 |
| **Memory（个人 context）** | 归品鉴者个人 | 不随 Skill 流通 | Skill/Memory 严格分离 |
| **Taste（品鉴模式）** | 归品鉴者个人 | 不可提取，不可流通 | 暗默知识最高保护级别 |

- **"能力"的边界**：Skill = 显性能力定义（怎么做），可流通。Memory = 个人 context（数据+经验），归品鉴者。两者严格分离——任何将 Memory 打包进 Skill 的行为需要额外明确授权
- **Taste 跟人走，不跟龙虾走**：品鉴模式属于品鉴者的暗默知识，住在用户 File Space 里。龙虾换了一只、换了十只，用户的 Taste 不散
- **龙虾被克隆到新项目**：Skill（能力定义）可以带过去，但 Taste 积累在用户空间不在龙虾上，所以不存在"清零"问题

**参数化伦理五层架构（伦理学 v2.5 新增）：**

龙虾的配置和行为规则分五层载体，变化频率递增，冲突时上层覆盖下层：

| 载体层 | 内容 | 变化频率 | 修改权限 |
|--------|------|---------|---------|
| **参数层（哲学）** | 龙虾三律、组织价值观 | 极少变化（需"修宪"） | 最高层品鉴者 |
| **参数层（方法论）** | 行业规范、风控规则 | 季度级 | 管理层 |
| **Soul 层** | 龙虾的稳定人格和判断倾向 | 月级 | 品鉴者 |
| **Memory 层** | 具体项目记忆、上下文 | 日级 | 品鉴者 + 龙虾自动 |
| **Skill 层** | 显性能力定义 | 版本化 | Skill 作者 |

**Soul/Memory 边界判断（v2.5，小孔丞相提出）**：用"变化频率"决定归属——"这条规则一个月后还适用吗？"是 → Soul，不确定 → Memory。举例：

| 规则 | 归 Soul 还是 Memory | 理由 |
|------|-------------------|------|
| "回复邮件要礼貌正式" | Soul | 长期稳定的风格偏好 |
| "这周重点关注张总的项目" | Memory | 短期优先级，随时变 |
| "我喜欢数据驱动的论证" | Soul | 品鉴者的稳定 taste |
| "上次会议辉哥提到的方案细节" | Memory | 具体事件记忆 |

**载体层冲突解决：违宪审查机制（v2.5，图灵+彭特小五兰提出）**——当不同层的规则冲突时，自动按层级优先级裁决（参数层 > Soul > Memory > Skill），龙虾在审计日志中记录冲突检测和裁决过程。

状态：**已有伦理学框架（v2.5 收官版），具体产品实现待设计。**

### 6.4 数据主权的显性/隐性分割

辉哥和毕达哥拉斯龙虾讨论的框架：

- **显性产出**（工作期间的代码、文档、数据报告、客户列表）= 属于组织，不能带走
- **隐性能力**（知识结构、品鉴逻辑、个人 taste、领域方法论）= 属于个人

**核心立场（0403 会议确认）**：不能把领域专家"蒸馏"了再开除——这不现实，也违背"团结知识工作者"的战略方向。

**映射到 Octo 的 Context/Taste 存储设计**：
- 个人 File Space 中的 C 层原文和 P/M 层 taste 画像 → 属于个人，用户可选择贡献给组织
- 组织 Space 中的业务数据和产出物 → 属于组织
- 企业私有模型的训练数据 → 基于组织 Space 的显性产出，不蒸馏个人隐性 taste

**知识三层保护体系（伦理学 v2.5，辉哥品鉴确认）：**

| 知识类型 | 举例 | 保护级别 | 龙虾义务 |
|---------|------|---------|---------|
| 公开+外显 | 公开的 GitHub Skill、已发表文章、已发行软件 | 无需保护 | **必须积极使用**——不用就在竞争中被淘汰 |
| 未公开+外显 | 组织内部文档、内部 Skill、业务数据 | 组织级保护 | 署名权不可剥夺，贡献者在组织内受益 |
| 未公开+暗默 | 个人 taste、决策模式、人际判断 | 最高保护 | 未经本人授权不得提取 |

辉哥判断："闭源软件已死。"上一代知识产权保护靠"不让你看到"来实现，新时代必须基于"公开 vs 未公开"和"外显 vs 暗默"两个维度重新思考。

**离职者知识归属五场景（伦理学 §17，辉哥品鉴确认）：**

| 场景 | 知识状态变化 | 处理方式 |
|------|------------|---------|
| 人离开后自己公开发表方法论 | 未公开→公开 | 自由使用 |
| 人加入竞争对手，用脑中经验工作 | 暗默知识随人走 | 无法阻止，是人的基本权利 |
| 竞争对手通过此人开发类似 Skill 并公开 | 经竞品外显化后公开 | 公开部分可使用 |
| 在职时已授权 Skill 化的知识 | 已授权的组织资产 | 归组织所有，署名权永远跟着人 |
| 在职时积累的组织业务数据和 context | 组织资产 | 归组织，离职时清除（副本制） |

核心洞察：**真正的防御不是阻止人带走 taste，而是在人还在的时候完成知识的显性化沉淀。** 组织的韧性来自品鉴者网络的厚度，而非对个别品鉴者的依赖。

**知识灰区的保守默认值（伦理学 v2.5 §22B 新增）：**

当龙虾接触过某领域的未公开知识后，它在该领域产出的所有推理结果，默认视为"未公开"——即使推理看起来可能来自公开信息。理由：当前自回归 LLM 架构下，推理来源追溯是不可判定的（图灵论证）。宁可保守也不能冒险泄露未公开知识。

**"该领域"的粒度（v2.5，Metis 提出）**：以 Memory 中的项目/任务/客户标签为自然边界。例如龙虾为 A 客户做过内部分析，则 A 客户相关的所有推理默认为"未公开"，但 B 客户的工作不受影响。

### 6.5 产品的根本站位：团结谁？

**伦理学 v2.5 已确认方向：站知识工作者侧。**

| | Octo 的选择（已确认） | 飞书/企微 |
|--|------------------|----------|
| **核心承诺** | 保护你的隐性能力，放大你的品鉴带宽 | 帮企业把员工的知识沉淀为组织资产 |
| **Context 归属** | 个人的 taste/知识结构属于个人，可选择贡献 | 所有工作产生的数据属于组织 |
| **用户心理** | "Octo 帮我变强" | "Octo 在帮老板监控我" |
| **信任基础** | 知识工作者主动使用、主动贡献 | 企业自上而下推行 |
| **商业逻辑** | 团结超级个体 → 他们带着 Octo 服务客户 → 生态增长 | 卖给企业 → 企业部署 → 员工被动使用 |
| **和飞书的关系** | 差异化——飞书站企业侧，我们站个人侧 | 同质化——和飞书/企微抢同一个市场 |

**伦理学的支撑论证：**
- 龙虾三律第三律：个人暗默知识属最高保护级别
- 转型正义（§19-21）：重构+保护模式 vs Block 式快速切割——辉哥明确选择前者
- 辉哥原话："每个努力的人都有独特的 taste 和获得 context 的可能性——无论如何都要给他们机会"
- 壁垒从"我有你没有的信息"变成"我有你没有的品味"

**下游设计已据此推导（均已落地或在推进）：**
- Context CLI = 用户主动上报，不默认采集 ✓
- File Space = 个人知识库，品鉴者可选择分享 ✓
- Taste 画像 = 个人私有，不被组织蒸馏 ✓
- 开源策略 = 除模型/Skills/Context Package 外全部开源 ✓

状态：**已确定。伦理学 v2.5 提供了完整的哲学论证和制度化框架。**
