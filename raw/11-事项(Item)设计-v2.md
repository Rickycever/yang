# 事项（Item）设计 V2

## 概述

"事项"是 Octo 中与 Channel、Thread 同属产品基础结构的一等公民对象。它是**人与 Agent 共同读写的结构化工作单元**，是整个 Agent OS 协作协议的基石。

这份文档与以下文档衔接：

- `07-Octo与OpenClaw的关系.md`：两层 Agent 分工（品鉴型 / 执行型）
- `PRD-CmdK-品鉴反馈器.md`：品鉴信号的深反馈机制
- `05-Octo-Shell-UI概念方案-v4.md`：Shell UI 整体形态
- `01-Dual-Native-Design-v1.5-洁净版-tikz.md`：Dual-Native 架构

本文档定义"事项"这一对象的完整设计：定义、UI 形态、生命周期、层级结构、Agent 协议。

---

## 设计亮点（为什么这个设计特别巧）

如果只看一句：**"事项"不是一个功能，是一个标准**——它定义了"人 + 多层 Agent 协作"的标准化工作单元，Octo 是这个标准的第一个实现。以下 6 个结构性红利是这个判断的具体支撑。

### 产品内在红利（事项结构本身带来的）

**1. 人的品鉴反馈怎么回到 Agent？事项天然就有解**

之前一直在思考一个问题：人给了品鉴反馈、给了品鉴信号，这些怎么真正对 Agent 起作用？要单独做一套 feedback loop 吗？

好像通过"事项"这个事情，天然就有解了。

你的品鉴信号（通过 / 驳回 / 批注）就直接留在事项里，和执行记录并排存着。Agent 下次做类似的事，翻一下过去同类事项——就是在读你怎么评的、什么该做、什么不该做。不需要单独的偏好数据库，也不需要单独的训练流程——**事项本身就是 Taste 的采集现场和反馈通道**。

**2. 养得好的 OpenClaw，就是你的数字劳动力分身**

理论上，一只养得好的 OpenClaw——通过事项学会了你的 Taste（你怎么评、你的风格），通过 FileSpace 拿到了你的部分 Context（素材、know-how）——是有可能在**某些场景下**替你做出和你一样的判断 / 决策的。

OC 是你的，FileSpace 也是你的——**相当于帮你养了一只属于你的龙虾**。

这背后是一个明确的价值立场：**我们要保护知识劳动者**。OC 是你的，FileSpace 是你的，养出来的龙虾也是你的——换个公司，龙虾跟着你走。你投入的时间、建立起的品味、积累的 know-how，不会因为一次跳槽就蒸发，也不会被公司扣下当作"交接资产"。

"数字劳动力分身"这个词有了真正能落地的形态——它活在你自己的事项历史和 FileSpace 里，归你所有。

**3. 解开了原来那个关于"不能把用户当肉鸡"的悖论**

之前 Octo 的架构有个矛盾：

- OC 是用户的 —— 我们不能主动去抓 Runtime Agent 的具体执行信息（不能把用户当肉鸡）
- 但系统又需要知道 Agent 到底做了什么，才能支持协作、审计、再编排

事项给出的解法：**用户主动把信息放进事项**。Runtime Agent 把执行轨迹和产出（比如具体的 diff）写到事项的执行载荷里——这是用户知情、用户授权的共享，不是我们偷偷采集。

或者更轻量的场景：Runtime Agent 跑在用户本地，用户自己问自己的 OC "某某 Runtime Agent 改了什么（具体的 Diff）"——OC 读事项就能答。上报是用户自己发起的，不存在隐私问题。

顺带还带来一个合规红利：append-only 品鉴信号 + 执行轨迹 + 归属链 = 天然的审计流。金融、医疗、法律这些"AI 必须可追溯"的行业，Octo 的答案是**原生的**，不是为合规打补丁。

### 商业与生态红利（架构带来的）

**4. OpenClaw 贵，但不一定比 Runtime Agent 干得好——分工才是答案**

OpenClaw 的 Token 消耗是 Runtime Agent 的 5-10 倍。但 OC 干活的质量，不一定比 Runtime Agent 更好——它真正擅长的是**编排**（记住你的偏好、拆任务、决定派谁去做），不是亲自写代码或画图。

所以分工就自然了：

- **OC 做编排**（它的强项，Token 贵也值得）
- **Runtime Agent 做执行**（它的强项，Token 便宜还干得好）

这个架构的直接好处：用户已经订阅的 Claude Max / Codex Pro / Cursor，可以直接接进来当 Runtime Agent 用。**Octo 自己不烧最大的那份 Token**——它做的是调度和品鉴采集，单用户订阅能做得很薄。

**5. 个人能力 × 组织能力的天然分工**

- **OpenClaw = 个人资产**（付个人订阅；养好了是你的龙虾，员工离职跟着走）
- **Runtime Agent = 组织资产**（付团队订阅；团队里人人都能调用的共享运力）

一个 Agent Team 在 Thread / Channel 里运行就变得很自然——成员各自的 OC 负责编排，共享的 Runtime Agent 池负责执行。这是**对企业最友好的 SaaS 账本**：个人订阅随员工走，组织订阅是纯共享算力——可以集中采购、议价，不跟着员工变动。

**6. 能力的内生评级——基于真实工作的口碑**

每个 Runtime Agent 在事项里干过多少活、通过率多少——这是它的**口碑**。每只 OpenClaw 的代理品鉴准确率、驳回率——这是它的**能力画像**。

这是一个**基于真实工作的评级系统**——不是 benchmark 分数，不是营销材料。

企业内部选用哪个 Runtime Agent、派哪只龙虾，可以数据驱动。对外——未来 Runtime Agent 要进 Octo 生态，需要有"被事项考核过的履历"，而不只是模型发布会的 PR 文案。

## 1. 定义

**事项 = 一条被赋予"结构化骨架"的顶层消息。**

当一条消息挂上骨架，就从"聊天"变成了"工作"。骨架由**人和 Agent 共同读写**，承载 5 件事：

1. **目标**（要做什么）
2. **状态**（做到哪了）
3. **归属链**（谁派、谁做、谁评）
4. **品鉴信号流**（做得如何，append-only 的历史）
5. **执行载荷**（过程轨迹 + 产出物，含 diff / 文件 / 素材引用）

从展示归类上看，事项和 PDF、Markdown 等属于**右侧面板里的内容类型**——同一片展示区可以切换呈现它们。不同的是：事项比静态文档多承载了状态机、品鉴流、归属链这些协作语义。

---

## 2. 在 UI 中的三种形态

事项有三个面：对话流里是**卡片**，右侧面板是**详情**，还有一个 Channel / Thread 级别的**总览视图**。

**核心逻辑**：人布置一个事项 → OpenClaw 或 Runtime Agent 认领 → 认领方推进状态、填写执行载荷 → 品鉴者给品鉴信号 → 完成。

### 2.1 对话流中的卡片（轻量入口）

在 Channel 或 Thread 里，事项显示为一张**状态卡片**：

- **左**：状态徽章（待办 / 进行中 / 待品鉴 / 完成）
- **中**：标题 + 一句摘要
- **右**：认领人头像 + 快捷品鉴按钮（通过 / 驳回 / 批注）

对话流不被骨架侵占，卡片只给最关键的信息。

### 2.2 右侧面板的事项详情（与 PDF / Markdown 同级的展示类型）

**点卡片 → 右侧面板展开事项详情**。右侧面板本身是一个多形态展示区——同一个位置可以切换呈现 PDF、Markdown、事项详情、Thread——事项只是其中一种内容类型：

```
┌──────────────────────────────────────┐
│ ▌事项标题          【状态】【···】   │ ← 头部
├──────────────────────────────────────┤
│ 目标 / 描述                          │
│ 归属链：[人] 创建 → [龙虾] 认领 → [人] 待评 │
├──────────────────────────────────────┤
│ 执行载荷                             │
│ ├─ 轨迹（Agent 做了哪些步骤）        │ ← 主体
│ ├─ 产出（diff / 文件 / 图片 / 表格) │
│ └─ 引用素材（从 FileSpace 拿了什么） │
├──────────────────────────────────────┤
│ 品鉴信号流（append-only 历史）       │
│ ├─ [通过] 龙虾 A · 10:23             │ ← 底部
│ ├─ [批注] 人 · 10:25："第3段改一下"  │
│ └─ [驳回] 人 · 11:10："方向错了"     │
├──────────────────────────────────────┤
│ [通过 / 驳回 / 批注 / Cmd+K 深反馈]   │
└──────────────────────────────────────┘
```

### 2.3 事项总览视图（Channel / Thread 级别的盘点）

在 Channel 或 Thread 里可以切换到一个**事项总览视图**——呈现这个频道或子区里**有哪些事项**、**各被谁（人 / OpenClaw / Runtime Agent）认领了**、**当前什么状态**。这个视图大概率也开在右侧面板里，作为多形态展示区的另一种切换状态。

用户先看到一张"这个频道的事项地图"，点击某一项，进入 2.2 的详情视图看细节。

## 3. 生命周期

```
      ┌── blocked
      │
todo ──claim──→ in_progress ──submit──→ in_review ──approve──→ done
                     │                       │
                     ├─ cancelled            ├─ reject → in_progress
                     └─ trace/artifact       │
                        持续累积             └─ append 品鉴信号（永不丢失）
```

**核心约定**：

- **任何人 / Agent 都可标状态**，但 **claim 的那方负责推进**
- **Agent 默认只标到 `in_review`**，最终 `done` 由具备品鉴权的实体给出（人 或 被授权的代理龙虾）
- **品鉴信号 append-only**，改主意就追加新信号，不覆盖历史（Taste 完整可审计，对齐 Octo 伦理学 §P3）

---

## 4. 层级与容纳关系

- **Channel 可以容纳多个事项**（和消息、Thread 并列在 Channel 里）
- **Thread 也可以容纳多个事项**（和消息并列在 Thread 里）
- **两者不冲突**——Channel 顶层放"大的"事项，Thread 里放"围绕某个讨论收敛出来的"事项

事项自身也可以有**子事项**——一个事项拆成多个子事项，支持多层 Agent 编排：父事项由龙虾持有，子事项派给 Runtime Agent 或 Sub-Agent。这是 "Agent 任意深度编排" 在产品层的实体——Claude Code 的 Agent Teams、Codex 的 worker、自研 Runtime，都可以是某个事项下的某个节点。

## 5. 事项与其他对象的关系

```
Workspace (JCMA_AI)
└── Category (产品 / 研发 / ...)
    └── Channel (后端架构讨论)
        ├── 消息
        ├── 事项
        └── Thread（讨论空间）
            ├── 消息
            └── 事项
```

关键关系：

- Channel 和 Thread 都是容器，**都可以放事项和消息**
- **Thread 里挂事项**（Thread 是容器，事项是 Thread 里的内容之一）——不是反过来
- 事项的**执行载荷**里可以有若干 Artifact（diff / 文件 / 图片等产出物）——Artifact 归属于事项
- FileSpace 的素材被事项**引用**（按复制语义，引用时触发受控复制）

---

## 6. Agent 协议：事项是可观测通道

Runtime Agent 可能跑在用户本地、跑在 Octo 提供的 VM、甚至跑在云端——但无论跑在哪，**只要它把执行信息写进事项，用户就能看见**：

- Agent 认领事项后，边做边把**每一步轨迹**写进事项的执行载荷
- 做出来的**产出物**（代码 diff、文件、图片、表格）作为一个 artifact 附在事项上
- 做完后把事项标到"待品鉴"，交给人或代理龙虾评

这解决了远程 Agent 执行不可观测的问题——**事项是 Agent 和人之间唯一的回传通道**。

Diff 是产出物的一种——Agent 把 unified diff 附上去，前端做 diff viewer 渲染。未来图片 diff、数据表 diff、设计稿 diff，都是产出物类型的自然扩展。
